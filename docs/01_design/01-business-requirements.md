## 1. ドキュメント目的と適用範囲

### 1.1 目的
本ドキュメントは、ナンバーズ（Numbers 3/4）の当選番号を独自の「中国罫線」理論とAI技術を用いて予測するWebアプリケーションシステムの**ビジネス要件**を定義する。本書により、開発チーム全員が共通認識を持ち、実装・テスト・検証を効率的に進めることを目的とする。

### 1.2 適用範囲
- 対象システム: ナンバーズAI予測Webアプリケーション（以下「本システム」）
- 対象抽選: ナンバーズ3、ナンバーズ4
- 対象ユーザー: モバイルデバイスからアクセスする一般エンドユーザー
- 開発期間: MVP（Minimum Viable Product）として7日間で初期バージョンを完成

### 1.3 除外事項
本バージョンでは以下を対象外とする：
- ユーザー認証・会員管理機能
- 課金・決済機能
- データ分析ダッシュボード
- 管理者機能

## 2. ゴール定義

### 2.1 誰が
- **プライマリ**: 個人開発者（1名）
- **セカンダリ**: 将来的には複数のエンドユーザー

### 2.2 何を
独自の「中国罫線」理論に基づく予測表を自動生成し、AIモデルが過去約6,700回分のデータから学習したパターンを活用して、ナンバーズ3/4の当選番号候補をスコア順にランキング表示するWebアプリケーションを構築する。

### 2.3 どれくらいの時間で
- **MVP版**: 7日間で基本機能を実装し、Vercelにデプロイ
- **完全版**: MVP検証後、3ヶ月以内に全機能を段階的に実装

### 2.4 達成基準
- ユーザーが回号とリハーサル数字を入力すると、5秒以内にAI予測結果が表示される
- N3/N4、ボックス/ストレート、0なし/0ありの全パターンで予測が実行される
- モバイルデバイスで快適に操作できるUI/UXが実現される

## 4. プロジェクト概要

### 4.1 ビジョン
独自の「中国罫線」分析理論を、データサイエンスとAI技術によって客観的かつ再現可能な「予測科学」へと昇華させる、世界に一つだけのパーソナル意思決定支援システムを構築する。

### 4.2 プロジェクト背景
- 紙媒体で管理されていた「中国罫線」理論をデジタル化
- 過去約6,700回分の抽選データを活用
- 第4954回以降（約1,800回分）に存在するリハーサル数字を分析材料として追加
- AIによる多角的なパターン学習により予測精度を向上

**⚠️ 重要: データ基準点**

**すべてのデータモデルとAIモデルの基準点は以下の通りです：**
- **基準回号**: 第6758回
- **基準日**: 2025年6月30日
- **データ収集方法**: 第6758回を基準にさかのぼって対応する
- **適用範囲**: すべてのモデルのもとは上記に準ずる

この基準点は、システム全体のデータ一貫性と再現性を保証するために必須です。

### 4.3 対象システム
- **システム名**: ナンバーズAI予測システム（Numbers-AI）
- **システム種別**: Webアプリケーション（PWA対応）
- **対象デバイス**: スマートフォン（iOS/Android）、タブレット、PC
- **対象ブラウザ**: Chrome, Safari, Edge（最新版）

### 4.4 開発アプローチ
MVP（Minimum Viable Product）から段階的に機能を拡張するアジャイル開発を採用：

**MVP（Phase 1）**: 7日間（完了）
- N3/N4の基本予測機能
- 0なし（パターンA）のみ
- 直近100回分のデータで学習（完了後、4801回分から最新回までに拡大）
- CSVベースのデータ管理

**Phase 2**: MVP+2週間
- 学習データを1,800回分に拡大
- 予測精度の向上

**Phase 3**: MVP+1ヶ月
- 全6,700回分のデータで学習
- アンサンブル学習の実装

**Phase 4**: MVP+3ヶ月
- 0あり（パターンB）の実装
- 統一モデルの構築
- Supabase/GCP Cloud Runへの移行

## 5. ペルソナ定義

### 5.1 プライマリペルソナ: 予測分析ユーザー

**基本情報**
- 名前: 田中 健太（仮名）
- 年齢: 35-50歳
- 職業: 会社員
- ITリテラシー: 中〜高（スマートフォンアプリの日常的利用）

**利用動機**
- ナンバーズの当選確率を高めたい
- データに基づく客観的な予測を求めている
- 直感ではなく論理的な根拠が欲しい

**利用シーン**
- 抽選日の午前中（リハーサル数字公開後）
- 通勤時間や休憩時間にスマートフォンから利用
- 1回の利用時間: 3〜5分程度

**ニーズ**
- 素早く予測結果を確認したい
- 複雑な操作は避けたい
- 予測の根拠が知りたい
- 過去の履歴は不要（当日の予測のみ重要）

**ペインポイント**
- 複雑な理論を理解するのは難しい
- 時間がないのに操作が煩雑だと使わなくなる
- ログインなどの面倒な手続きは避けたい

## 6. ユーザーフロー

### 6.1 基本フロー（3ステップ）

```
[ユーザー起動]
     │
     ▼
┌──────────────────┐
│  ホーム画面       │
│  「新規予測開始」 │
└──────────────────┘
     │ タップ
     ▼
┌──────────────────┐
│ Step 1: 回号入力  │
│  4桁の数字入力    │
│  例: 6701        │
└──────────────────┘
     │「次へ」
     ▼
┌──────────────────┐
│ Step 2:          │
│ リハーサル数字入力│
│  N4: [3][7][8][2]│
│  N3: [1][4][9]   │
└──────────────────┘
     │「AI予測実行」
     ▼
┌──────────────────┐
│ AI分析中         │
│  ローディング     │
│  3〜5秒         │
└──────────────────┘
     │
     ▼
┌──────────────────┐
│ Step 3: 予測結果  │
│                  │
│ [N3] [N4] タブ   │
│ [ボックス][ストレート]│
│ [軸数字][総合]   │
│                  │
│ ランキング表示    │
└──────────────────┘
     │
     ▼
[ユーザーが結果確認]
```

### 6.2 詳細ユーザーアクション

**Step 1: 回号入力**
1. ユーザーがホーム画面の「新規予測を開始」をタップ
2. 回号入力画面が表示される
3. 数字キーボードで4桁の回号を入力
4. 「次へ」ボタンをタップ

**Step 2: リハーサル数字入力**
1. N4の入力フィールドに4桁の数字を入力
2. N3の入力フィールドに3桁の数字を入力
3. 各桁は自動的に次のフィールドにフォーカス移動
4. 「AI予測を実行」ボタンをタップ

**Step 3: 結果表示**
1. AIが分析を実行（3〜5秒）
2. 予測結果画面が表示される
3. デフォルトは「N3 / ボックス / 軸数字」モード
4. ユーザーはタブを切り替えて各種予測を確認
5. 軸数字候補をタップして展開・詳細を確認
6. 手動指定軸で独自の分析も可能

### 7.2 機能詳細仕様

#### F-01: 回号入力機能

**目的**: ユーザーが予測対象の抽選回を指定する

**入力**
- 回号（4桁の数字、例: 6701）

**処理**
- 数値のみ入力可能
- 4桁に満たない場合は「次へ」ボタンを無効化
- 入力中はリアルタイムで桁数を表示

**出力**
- 入力された回号をセッションに保存
- リハーサル数字入力画面へ遷移

**バリデーション**
- 必須入力
- 4桁の数字（0000〜9999）
- 数字以外の入力は不可

**エラーハンドリング**
- 無効な入力時はエラーメッセージを表示
- 「回号は4桁の数字で入力してください」

---

#### F-02: リハーサル数字入力機能

**目的**: AI予測の重要な手がかりとなるリハーサル数字を入力する

**入力**
- N4リハーサル数字（4桁、千百十一の位）
- N3リハーサル数字（3桁、百十一の位）

**処理**
- 各桁は独立した入力フィールド
- 数字入力時に自動的に次のフィールドへフォーカス移動
- Backspaceキーで前のフィールドへ戻る
- モバイルでは数字キーボードを自動表示

**出力**
- 入力されたリハーサル数字をセッションに保存
- AI予測実行をトリガー

**バリデーション**
- N4: 4桁すべて必須
- N3: 3桁すべて必須
- 各桁は0〜9の数字

**UI/UX要件**
- 視覚的に桁位置が明確（千、百、十、一のラベル表示）
- 入力済みの桁は視覚的に区別
- フォーカス中のフィールドはハイライト

## 8. 非機能要件

### 8.1 パフォーマンス要件

| 項目 | 目標値 | 測定条件 |
|------|--------|----------|
| 初期ローディング時間 | 3秒以内 | 初回アクセス時 |
| 画面遷移時間 | 500ms以内 | ページ間遷移 |
| AI予測応答時間 | 5秒以内 | 予測実行ボタン押下から結果表示まで |
| 予測表生成時間 | 100ms以内 | 1表あたり |
| APIレスポンス時間 | 3秒以内 | 95パーセンタイル |

### 8.2 ユーザビリティ要件

**モバイルファースト設計**
- 最小画面サイズ: 320px × 568px（iPhone SE）
- タッチターゲット最小サイズ: 48px × 48px
- フォントサイズ: 最小14px（本文）、最小12px（補助テキスト）
- コントラスト比: WCAG 2.1 AA準拠（4.5:1以上）

**操作性**
- 3ステップで予測完了
- 各ステップは1画面に収まる情報量
- 戻る操作が可能
- エラーメッセージは具体的で理解しやすい

**アクセシビリティ**
- セマンティックHTMLの使用
- ARIAラベルの適切な設定
- キーボード操作対応（デスクトップ）
- スクリーンリーダー対応

### 8.3 互換性要件

**ブラウザサポート**
- Chrome 90+
- Safari 14+
- Edge 90+
- Firefox 88+

**デバイスサポート**
- iOS 14+
- Android 10+
- タブレット（iPad, Android tablet）
- デスクトップPC（Windows, macOS, Linux）

**画面解像度**
- スマートフォン: 375px〜428px
- タブレット: 768px〜1024px
- デスクトップ: 1280px〜

### 8.4 セキュリティ要件

**通信セキュリティ**
- HTTPS必須（TLS 1.2以上）
- セキュアCookie使用
- CORS設定の適切な管理

**データ保護**
- 個人情報は収集しない（MVP版）
- セッションデータはブラウザのLocalStorageに保存
- APIキーは環境変数で管理

**脆弱性対策**
- XSS対策（入力値のサニタイズ）
- CSRF対策（トークン使用）
- SQLインジェクション対策（プリペアドステートメント）

### 8.5 可用性要件

**稼働率**
- 目標: 99.0%（MVP版）
- 計画メンテナンス: 月1回、深夜帯

**障害復旧**
- 目標復旧時間（RTO）: 24時間以内
- 目標復旧ポイント（RPO）: 24時間以内

**バックアップ**
- データベース: 日次自動バックアップ（Phase 4以降）
- コード: Gitリポジトリで管理

### 8.6 拡張性要件

**データ量の増加対応**
- 当面の想定: 約6,700回分のデータ
- 年間増加: 約200回分
- 10年後も同等のパフォーマンスを維持

**機能拡張対応**
- 新パターン（パターンC, D等）の追加が容易
- 新特徴量の追加が容易
- 新AIモデルの組み込みが容易

**スケーラビリティ**
- 同時接続ユーザー数: 100人（MVP版）
- 将来的に1,000人に拡張可能な設計

### 8.7 保守性要件

**コード品質**
- TypeScript（フロントエンド・バックエンド）
- ESLint + Prettierでコードスタイル統一
- コメント率: 10%以上
- 複雑度: Cyclomatic Complexity 10以下

**テストカバレッジ**
- ユニットテスト: 80%以上（Phase 2以降）
- 統合テスト: 主要機能100%
- E2Eテスト: クリティカルパス100%

**ドキュメント**
- README: 環境構築手順
- API仕様書: OpenAPI 3.0形式
- コンポーネント仕様: Storybookで管理

## 9. 制約条件

### 9.1 技術的制約

**開発言語・フレームワーク**
- フロントエンド: Next.js 14+ (TypeScript)
- バックエンド: Next.js API Routes または FastAPI（Phase 4）
- AI/ML: Python 3.10+, XGBoost, scikit-learn
- データベース: CSV（MVP）→ Supabase PostgreSQL（Phase 4）

**インフラ制約**
- ホスティング: Vercel（MVP）
- AI API: GCP Cloud Run（Phase 4）
- ストレージ: Vercel Blob Storage または Supabase Storage

**外部サービス制約**
- データソース: みずほ銀行公式サイト、特定Webサイト
- 更新頻度: 手動更新（MVP）→ 自動スクレイピング（Phase 2）

### 9.2 リソース制約

**開発体制**
- 開発者: 1名（フルスタック）
- 期間: MVP 7日間
- 予算: 無料枠内（Vercel, Supabase等）

**データ制約**
- MVP版: 直近100回分のデータ
- Phase 2: リハーサルあり約1,800回分
- Phase 3以降: 全約6,700回分

### 9.3 ビジネス制約

**法的制約**
- ギャンブル関連法規の遵守
- 公営くじの範囲内
- 予測は「参考情報」として提供（保証なし）

**免責事項**
- 当選を保証するものではない
- 予測結果の利用は自己責任
- データの正確性は保証しない
