## 1. ドキュメント目的と適用範囲

### 1.1 目的
本ドキュメントは、ナンバーズAI予測システムの開発を、MVP（Minimum Viable Product）から完全版まで段階的に進めるための**実装計画**を定義する。特に、「**いつ、どの技術を導入するか**」と「**なぜその技術が今必要なのか（または不要なのか）**」を明確にすることで、開発者が迷わず最適な技術選択をできるようにする。

### 1.2 このドキュメントが解決する問題
開発中に以下のような疑問が生じることを防ぐ：
- 「なぜMVPではSupabaseを使わないのか？」
- 「Weights & Biasesはいつ導入すべきか？」
- 「Dockerコンテナ化はどのタイミングが適切か？」
- 「今、この技術的負債を作ることは許容されるのか？」

### 1.3 対象読者
- フルスタックエンジニア（開発実装者）
- 技術リード（技術選定判断者）
- プロジェクトオーナー（進捗管理者）

### 1.4 関連ドキュメント
- **docs/requirements.md**: 何を作るか（What）
- **docs/specifications.md**: どう作るか（How）
- **本ドキュメント**: いつ、どの順番で作るか（When & Action）

## 2. 実装戦略の全体像

### 2.1 段階的開発の哲学

本プロジェクトは、「**最小の投資で最大の学びを得る**」という原則に基づき、以下の段階を踏む：

```
MVP (Phase 1)
  │ 目的: コア理論の技術的実証
  │ 期間: 7日間
  │ 成果: 動くプロトタイプ
  ▼
Phase 2
  │ 目的: データ管理の堅牢化
  │ 期間: MVP + 2週間
  │ 成果: スケーラブルなデータ層
  ▼
Phase 3
  │ 目的: AI精度の科学的改善
  │ 期間: MVP + 1ヶ月
  │ 成果: 実験管理プロセスの確立
  ▼
Phase 4
  │ 目的: 本番運用可能なアーキテクチャ
  │ 期間: MVP + 3ヶ月
  │ 成果: マイクロサービス化
```

### 2.2 各フェーズの成功基準

| フェーズ | 成功基準 | 検証方法 |
|---------|---------|---------|
| **MVP** | 予測機能が動作し、結果が表示される | 手動テストで5回の予測が成功 |
| **Phase 2** | 1,800件のデータを安定して管理できる | SQLクエリのレスポンスタイムが1秒以内 |
| **Phase 3** | AIモデルの精度が5%向上する | W&Bダッシュボードで実験結果を比較 |
| **Phase 4** | 100人の同時接続に耐えられる | 負荷テストで99%のリクエストが5秒以内 |

## 3. MVP実装計画（Phase 1）

### 3.1 MVPの目的
**「中国罫線理論に基づくAI予測」というコンセプトが、技術的に実現可能であることを最速で証明する。**

### 3.2 MVP期間
**7日間**（2025年11月3日 〜 2025年11月9日）

### 3.3 MVP導入技術スタック

#### 採用する技術

| 技術 | 用途 | 採用理由 |
|------|------|---------|
| **Next.js 14** | フロントエンド + API Routes | フルスタック開発を単一フレームワークで完結でき、学習コストとセットアップ時間を最小化 |
| **TypeScript** | 型安全な開発 | バグの早期発見とリファクタリングの安全性を担保 |
| **React 18** | UI構築 | Next.jsのデフォルトUIライブラリ |
| **Tailwind CSS + shadcn/ui** | スタイリング | 高速なUI開発とモダンなデザイン |
| **Python (FastAPI)** | AI推論API | TypeScriptでのAI実装は困難なため、PythonをAPI化 |
| **XGBoost** | AIモデル | 表形式データに対して高精度かつ学習が高速 |
| **scikit-learn** | 前処理・評価 | Pythonの標準的なMLライブラリ |
| **CSV + JSON** | データストレージ | 100件程度のデータなら、DB不要で高速に動作 |
| **Vercel** | ホスティング | Next.jsとの親和性が高く、無料枠で十分 |

#### 採用しない技術とその理由

| 技術 | 採用しない理由 |
|------|---------------|
| **Supabase (PostgreSQL)** | MVPのデータ量（100件）では、CSV管理の方がセットアップコストがなく高速。データベースのスキーマ設計、マイグレーション管理、接続管理といった複雑性を導入する必要がない。 |
| **Weights & Biases** | MVPの目的は「動くモデルを1つ作る」ことであり、多数の実験を比較管理する必要がない。W&Bのセットアップと学習コストは、この段階では投資対効果が低い。 |
| **GCP Cloud Run / Docker** | Vercelのサーバーレス関数で十分に機能し、インフラの複雑性を最小限に抑えるべき。コンテナ化は、リソース制限に直面してから検討すれば十分。 |
| **Redux Toolkit** | MVPでは状態管理の複雑性が低く、React標準のuseStateとuseContextで十分。Redux導入は、状態の複雑化が問題になってから検討。 |
| **Playwright (E2E)** | MVPでは手動テストで十分。自動テストのセットアップコストは、機能の安定化後に投資すべき。 |

### 4.3 Phase 2で導入する主要技術

#### Supabase (PostgreSQL)

**導入理由:**

1. **データ量の増加対応**
   - 学習データが100件 → 1,800件に増加
   - CSVファイルでは以下の問題が顕在化：
     - ファイルI/Oのオーバーヘッド増大
     - データの整合性担保が困難（同時書き込み、ロック管理）
     - 検索性能の低下（フィルタリング、ソート処理）

2. **リレーショナルデータの管理**
   - 過去実績データと予測結果の関連付け
   - 将来的なユーザー認証機能の導入準備

3. **バックアップとリストア**
   - Supabaseの自動バックアップ機能により、データ損失リスクを低減
   - CSVファイルは手動バックアップが必要で、ヒューマンエラーのリスクが高い

4. **開発効率の向上**
   - SQLクエリによる柔軟なデータ取得
   - Supabase Studioによる視覚的なデータ管理

**導入タイミング:**
- ~~MVPでCSV管理の限界を体感してから導入することで、データベースの価値を実感できる~~（**変更**）
- **最新の方針（2025-01-XX）**: AI精度改善（Phase 3）が完了してから実施予定
- 現在は学習データ範囲の拡大（1000回分→4801回分）を優先し、CSVベースの管理を継続
- 「なぜCSVではダメなのか」を経験的に理解した上での移行は、学習効果が高い

## 5. Phase 3実装計画

### 5.1 Phase 3の目的
**AIモデルの改善プロセスを、勘や手作業から、科学的で再現性の高いものへと進化させる。**

**⚠️ 最新の進捗状況（2025-01-XX更新）:**
- Phase 2とPhase 3を並行進行中
- **Phase 1（ベースライン確立）**: 
  - ✅ 1000回分でベースラインモデル作成完了
  - ⏳ 4801回分でベースラインモデル作成進行中
- **重要な方針決定**:
  - 特徴量エンジニアリング実験はスキップ（全特徴量を使用）
  - 1000回分での分割モデルは検討しない（統合モデルのみ）
  - 4801回分での分割モデルは条件付き（目標未達かつパターン間の精度差が大きい場合のみ）

### 5.2 Phase 3期間
**MVP + 1ヶ月**（2025年11月24日 〜 2025年12月23日）

**⚠️ 実際の進捗**: Phase 2とPhase 3を並行進行中。ベースライン確立（Phase 1）が進行中。

### 5.3 Phase 3で導入する主要技術

#### Weights & Biases (W&B)

**導入理由:**

1. **実験管理の必要性**
   - Phase 3では、AIの精度を0.1%単位で改善するために、以下のような多数の実験を実施：
     - ~~特徴量セットの比較（形状特徴のみ vs 全特徴量）~~（**スキップ**: 全特徴量を使用）
     - **ハイパーパラメータの最適化**（学習率、深さ、正則化）← **集中実施**
     - データセットの比較（1000回分 vs 4801回分）← **実際の進捗に合わせて変更**
     - **アンサンブル学習の検証**（チャート・マスターモデル vs リハーサル・コンテキストモデル）
       - リハーサルなしデータ（約5,000回分）で学習したモデル（Phase 4以降）
       - リハーサルありデータ（約1,800回分）で学習したモデル（Phase 4以降）
       - 両者のアンサンブル手法（重み付き平均）の最適化（Phase 4以降）
   - これらの実験を手動で管理すると：
     - どの設定で、どの精度が出たか忘れる
     - 実験の再現ができない
     - 過去の実験結果との比較が困難

2. **データドリブンな意思決定**
   - W&Bのダッシュボードで、全実験のAUC-ROC、Precision、Recallを可視化
   - 「このパラメータセットが最も精度が高い」という根拠を、データで示せる

3. **チーム開発への準備**
   - 将来的に複数人で開発する場合、実験結果の共有が容易

**導入タイミング:**
- Phase 2でデータ基盤が安定してから導入（**進行中**）
- 多数の実験を実施する準備が整った段階

   - モデル保存名にデータセット名を含める: `{dataset_name}_{target}_{type}.pkl`

4. **チャート・マスターモデルの構築（リハーサルなし）**

   - 各モデルの評価結果をW&Bに記録
   - 実行順序:
     ```bash

     # 03_feature_engineering.ipynb を実行（dataset_name='no_rehearsal_5000'）
     # → no_rehearsal_5000_*_data.pkl を生成
     # 04_model_training.ipynb を実行（dataset_name='no_rehearsal_5000'）
     ```

5. **リハーサル・コンテキストモデルの構築（リハーサルあり）**
   - リハーサルありデータ（約1,800回分）で6モデルを学習
   - ファイル命名規則: `rehearsal_1800_{target}_{type}.pkl`
   - 各モデルの評価結果をW&Bに記録
   - 実行順序:
     ```bash
     # 01_data_preparation.ipynb を実行 → rehearsal_1800.csv を生成
     # 03_feature_engineering.ipynb を実行（dataset_name='rehearsal_1800'）
     # → rehearsal_1800_*_data.pkl を生成
     # 04_model_training.ipynb を実行（dataset_name='rehearsal_1800'）
     # → rehearsal_1800_*.pkl を生成
     ```

6. **データセット設定の統一化**
   - `notebooks/dataset_config.py`を作成
   - データセット設定を一元管理（`DATASET_CONFIGS`）
   - フィルタリング関数、期待データ数、説明を定義

7. **アンサンブル手法の検証**
   - 推論時にリハーサル数字の有無を判定する機能を実装
   - リハーサルありの場合: 2つのモデルの出力を統合（重み付き平均）
     - 重み付き平均: `p_final = w1 * p1 + w2 * p2`
     - 検証データでAUC-ROCを最大化する重みを探索（グリッドサーチまたはベイズ最適化）
   - リハーサルなしの場合: チャート・マスターモデルのみ使用
   - 各モデルタイプ（軸数字/組み合わせ、N3/N4）ごとに最適重みを決定
   - アンサンブル手法の評価結果をW&Bに記録

**成果物:**
- チャート・マスターモデル（6モデル、約5,000回分で学習）
- リハーサル・コンテキストモデル（6モデル、約1,800回分で学習）
- アンサンブル重みの最適値（各モデルタイプごと）
- アンサンブル手法の評価レポート（単一モデルとの比較）

### 5.5 Phase 3成果物

| 成果物 | 説明 |
|--------|------|
| **W&Bプロジェクト** | 100以上の実験結果を記録 |
| **改善されたAIモデル** | AUC-ROC 5%向上を達成 |
| **特徴量分析レポート** | 重要な特徴量とその影響度 |
| **最適パラメータドキュメント** | 再現可能な最適設定 |
| **アンサンブルモデル** | チャート・マスターモデル（6モデル）とリハーサル・コンテキストモデル（6モデル） |
| **アンサンブル重み設定** | 各モデルタイプごとの最適重み値 |

## 6. Phase 4実装計画

### 6.1 Phase 4の目的
**AIエンジンとフロントエンドを物理的に分離し、それぞれを独立してスケール・更新できる、堅牢なマイクロサービスアーキテクチャへと移行する。**

### 6.2 Phase 4期間
**MVP + 3ヶ月**（2025年12月24日 〜 2026年2月2日）

### 6.3 Phase 4で導入する主要技術

#### GCP Cloud Run + Docker

**導入理由:**

1. **Vercelサーバーレス関数の限界**
   - 実行時間制限: 最大10秒（Hobby）、60秒（Pro）
   - メモリ制限: 最大1GB（Hobby）、3GB（Pro）
   - パッケージサイズ制限: 50MB
   - Phase 3で以下の問題が顕在化する可能性：
     - 複雑なアンサンブルモデルのロード時間が長い
     - NumPy/Pandas/XGBoostの依存関係が大きい
     - 推論処理が10秒を超える場合がある

2. **AIエンジンの独立性**
   - フロントエンド（Next.js on Vercel）とバックエンド（FastAPI on Cloud Run）を分離
   - AIモデルの更新時、フロントエンドに影響を与えない
   - AIエンジンのスケーリングを独立して制御可能

3. **リソースの柔軟性**
   - Cloud Runは、メモリ4GB、実行時間60分まで対応
   - GPUインスタンスへの移行も容易

**導入タイミング:**
- Phase 3でAIモデルが複雑化してから導入
- Vercelの制限に実際に直面してから移行

#### パターンB（0あり）の実装

**導入理由:**
- MVP〜Phase 3で、パターンA（0なし）の精度を十分に高めた後、パターンBに着手
- 両パターンの予測結果を統合し、ユーザーに最適な選択肢を提示

### 6.4 Phase 4実装タスク

#### Week 1-2: Dockerコンテナ化
- FastAPI AIエンジンのDockerfile作成
- ローカルでのDocker環境テスト
- Cloud Runへのデプロイ設定

#### Week 3-4: パターンB実装
- パターンBの予測表生成アルゴリズム実装
- パターンB用AIモデルの学習
- フロントエンドでのパターンA/B切替UI実装

#### Week 5-6: マイクロサービス統合
- Next.js ↔ Cloud Run間のAPI通信設定
- 認証・認可の実装
- パフォーマンステストと最適化

### 6.5 Phase 4成果物

| 成果物 | 説明 |
|--------|------|
| **Dockerコンテナ** | FastAPI AIエンジンのコンテナ |
| **GCP Cloud Run環境** | 本番運用可能なマイクロサービス |

## 7. 技術選定の判断基準

### 7.1 技術導入の意思決定フレームワーク

新しい技術を導入する際は、以下のチェックリストに基づいて判断する：

#### ✅ 導入すべき条件（3つ以上満たす場合）

1. **現在の技術で明確な問題が発生している**
   - 例: CSVの読み込みが遅い、Vercelの実行時間制限に引っかかる

2. **問題解決の効果が定量的に測定できる**
   - 例: W&B導入で、実験の比較工数が90%削減される

3. **学習コストが1週間以内**
   - 例: Supabaseの基本操作は、公式ドキュメントで2日で習得可能

4. **将来的な拡張性が見込める**
   - 例: PostgreSQLに移行すれば、将来のユーザー認証機能も実装可能

5. **無料枠または低コストで試せる**
   - 例: Supabase, W&B, GCP Cloud Runはすべて無料枠あり

#### ❌ 導入すべきでない条件（1つでも該当する場合）

1. **現在の技術で問題が発生していない**
   - 例: CSVで十分動いているのに、「将来のため」だけでDBを導入

2. **効果が不明確または測定不可能**
   - 例: 「なんとなく良さそう」という理由での技術導入

3. **学習コストが高い（1週間以上）**
   - 例: KubernetesやTerraformは、MVPには過剰

4. **既存の技術で代替可能**
   - 例: Redux Toolkitが不要な状態で、MobXを導入

5. **コストが予算を超える**
   - 例: 月額1万円以上のツールは、MVP段階では避ける

### 7.2 技術的負債の許容基準

以下の技術的負債は、MVP〜Phase 2では**意図的に許容する**：

| 技術的負債 | 許容理由 | 解消タイミング |
|-----------|---------|--------------|
| **E2Eテストなし** | 手動テストで十分。自動テストのセットアップコストが高い | Phase 3以降 |
| **CSVデータ管理** | 100件程度なら問題なし | Phase 2でSupabaseに移行 |
| **ハードコーディング** | 一部の定数はコード内に直書きしても、変更頻度が低ければ許容 | Phase 3で設定ファイル化 |
| **エラーハンドリングが簡易** | MVPでは全エラーを共通メッセージで表示 | Phase 3で詳細化 |
| **パフォーマンス最適化なし** | 100人以下なら問題なし | Phase 4で負荷テスト後に最適化 |

## 9. 技術的負債の管理方針

### 9.1 技術的負債の3段階分類

本プロジェクトでは、技術的負債を以下の3段階に分類し、管理する：

#### Tier 1: 許容される負債（MVP〜Phase 2）
- **例**: E2Eテストなし、エラーハンドリング簡易、CSS最適化なし
- **方針**: 機能の実現を優先し、品質は後で改善
- **解消タイミング**: Phase 3以降

#### Tier 2: 早期に解消すべき負債（Phase 2）
- **例**: CSVデータ管理、ハードコーディング、手動デプロイ
- **方針**: データ量やユーザー数の増加に備え、Phase 2で解消
- **解消タイミング**: Phase 2

#### Tier 3: 絶対に作ってはいけない負債
- **例**: セキュリティ脆弱性、データ損失リスク、法的問題
- **方針**: 初日から対策必須
- **解消タイミング**: 即座

### 9.2 技術的負債の記録方法

- **GitHub Issues**: 技術的負債をissueとして記録し、ラベル `tech-debt` を付与
- **優先度**: `p1-critical`, `p2-high`, `p3-medium`, `p4-low` で分類
- **解消計画**: 各フェーズのマイルストーンに関連付け

## 10. まとめ

### 10.1 実装計画の要点

本実装計画書により、以下が明確になった：

1. **MVPは最小限の技術で7日間で完成させる**
   - Next.js, TypeScript, CSV, XGBoost, Vercel

2. **Phase 2でデータ管理をSupabaseに移行する**
   - 理由: 1,800件のデータを安定して扱うため

3. **Phase 3でW&Bを導入し、AI精度を科学的に改善する**
   - 理由: 多数の実験を体系的に管理するため

4. **Phase 4でCloud Runに移行し、マイクロサービス化する**
   - 理由: Vercelの制限を超え、AIエンジンを独立してスケールするため

5. **技術的負債は3段階で管理し、意図的に作る負債と作ってはいけない負債を明確にする**

### 10.2 開発者へのメッセージ

この実装計画書は、あなたが「今、何をすべきか」「なぜこの技術を使うのか」を迷わず判断できるようにするためのガイドです。

**重要な原則:**
- ✅ 今必要な技術のみを導入する
- ✅ 問題が発生してから解決策を導入する（先読みしすぎない）
- ✅ 技術的負債は意図的に作り、計画的に解消する
- ✅ 各フェーズの成功基準を満たしてから次へ進む

それでは、素晴らしいプロダクトの開発を楽しんでください！🚀
