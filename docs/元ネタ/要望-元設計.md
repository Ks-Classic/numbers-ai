### **プロジェクト『PN-Analyzer』公式開発ドキュメント**

**はじめに：本ドキュメントの目的と構成**
本ドキュメントは、ナンバーズ予測AIシステム『PN-Analyzer』の開発に関する、唯一の公式な設計・計画書である。本プロジェクトは、独自の分析理論に基づき、AIを用いてナンバーズの当選番号を予測する、高度なWebアプリケーションを構築することを目的とする。

本ドキュメントは、以下の3部から構成される。
1.  **DOC-01-REQ：要件定義書:** システムが「何を」実現すべきかを定義する。
2.  **DOC-02-ARCH：システム設計書:** システムを「どのように」構築するかを定義する。
3.  **DOC-03-IMPL：実装計画書:** 最終的なゴールを見据えつつ、最短で価値を実証するための具体的な開発ステップ（MVP計画）を定義する。

### **DOC-01-REQ：要件定義書 (完全版 v5.0)**

#### **1.0 プロジェクト概要**
*   **1.1 ビジョン:** 独自の分析理論を、データサイエンスとAIによって客観的かつ再現可能な「予測科学」へと昇華させる、世界に一つだけのパーソナル意思決定支援システムを構築する。
*   **1.2 最終目的:** 利用者がリハーサル数字を入力することで、AIがその日の当選確率が高い「軸数字」を提案。利用者はその提案に基づき軸を選択し、最終的にAIがその軸を含む「ストレート」および「ボックス」の当選予測番号を的中確率順に提示。利用者はその情報を基に、実際に当選を勝ち取ることを最終目的とする。
*   **1.3 対象:** ナンバーズ3、ナンバーズ4

#### **2.0 機能要件**
*   **2.1 データ基盤構築機能（オフライン処理）**
    *   `F-DB-01`: **罫線データ・デジタル化:** 紙媒体の罫線ルールを、プログラムで利用可能なJSON形式(`keisen_master.json`)に変換する。
    *   `F-DB-02`: **過去実績データ・自動収集:** 過去全開催回の「当選番号」「リハーサル数字」をWebから自動収集し、CSV形式(`past_results.csv`)で出力する。
    *   `F-DB-03`: **予測表・一括生成:** 過去全回分の予測表（パターンA/B）をアルゴリズム通りに生成し、データベースに保存する。

*   **2.2 ユーザー向けアプリケーション機能**
    *   `F-APP-01`: **ユーザー認証:** Emailとパスワードによるログイン機能。
    *   `F-APP-02`: **新規予測セッション:** 利用者が抽選回、リハーサル数字（N3/N4）を入力し、分析を開始する機能。
    *   `F-APP-03`: **AIインタラクティブ予測:**
        1.  **軸数字提案:** AIが予測した「軸数字候補」を、スコア順にリスト表示する。
        2.  **軸数字選択:** 利用者が提示された軸数字から、任意の一つまたは複数をチェックボックス等で選択する。
        3.  **最終予測提示:** 利用者の選択をトリガーに、AIが最終的な「ボックス予測」「ストレート予測」の候補を、スコア順にリスト表示する。
    *   `F-APP-04`: **結果の可視化:**
        *   当選候補を選択すると、予測表上の該当数字がハイライト表示される。
        *   各候補の「出所（0なし/0あり）」ラベルを表示する。

*   **2.3 AIモデル機能（システム内部要件）**
    *   `F-AI-01`: **二段階予測アーキテクチャ:** 「軸数字予測モデル」と「組み合わせ予測モデル」で構成される。
    *   `F-AI-02`: **アンサンブル学習:** リハーサル数字の有無で学習データを分けた「チャート・マスターモデル」と「リハーサル・コンテキストモデル」の予測結果を加重平均して、最終的なスコアを算出する。
    *   `F-AI-03`: **統一モデル:** 「0なし(A)」「0あり(B)」の表の違いを、AI自身が特徴量として学習する単一のモデルを構築する。
    *   `F-AI-04`: **モデル分離:** N3/N4、ボックス/ストレートの予測モデルは、それぞれ独立して構築・運用する。

#### **3.0 非機能要件**
*   **3.1 パフォーマンス:** ユーザー操作に対するAIの応答時間は5秒以内。
*   **3.2 UI/UX:** モバイルファーストで設計された、機能性優先のシンプルなインターフェース。
*   **3.3 拡張性:** 新たな「表パターン」や「特徴量」の追加に、容易に対応できる柔軟な設計とする。

### **DOC-02-ARCH：システム設計書 (完全版 v5.0)**

#### **1.0 システムアーキテクチャ**
開発速度、低コスト、拡張性を最優先し、モダンなサーバーレス・マイクロサービスアーキテクチャを採用する。
*   **① ユーザー・インターフェース (Vercel):** Next.js(TypeScript)で構築。
*   **② データ・ストア (Supabase):** PostgreSQLデータベース。
*   **③ AIエンジン (GCP Cloud Run):** Python(FastAPI)で構築された、表生成とAI予測を実行する独立APIサーバー。
*   **④ AI実験環境 (Google Colab):** AIモデルの学習・評価・改善を行うオフライン環境。

#### **2.0 データモデル（データベース設計）**
*   **`keisen_master`:** 罫線データを格納。
*   **`past_results`:** 過去実績を格納。
*   **`generated_charts`:** 生成済み予測表を格納。
*   **`prediction_history`:** ユーザーの予測履歴を格納。

#### **3.0 表生成アルゴリズム仕様（最終確定版）**
*   **3.1 入力:** `round_number`, `pattern_type`
*   **3.2 Step 1: 予測数字の抽出:** 「前回」「前々回」の当選番号をキーに、`keisen_master`から対応する`predicted_digits`を全桁分抽出。
*   **3.3 Step 2: 「元数字リスト」の作成:**
    *   全`predicted_digits`を結合し、`pattern_type`に応じて「欠番」または「0」を追加し、昇順ソートする。
*   **3.4 Step 3: メイン行の組み立て:**
    1.  元数字リストが空になるまでループ。
    2.  リスト内のユニークな数字が4種類以上の場合、先頭4種類の数字を1つずつ抜き出して4要素の行を作成。
    3.  4種類未満の場合、ユニークな数字をベースとし、要素数が4つになるまでその中の**最大値で不足分を埋め（パディングし）**、4要素の行を作成。
    4.  作成した行の数字を元数字リストから削除。
*   **3.5 Step 4: 最終表の構築:**
    1.  メイン行を偶数行・偶数列に初期配置。
    2.  **パターンBの場合:** 表の中心の空きマス（左上優先）に`0`を配置。
    3.  **裏数字（縦パス）:** グリッドに変化がなくなるまで、`grid[y+1][x] = (grid[y][x] + 5) % 10` を適用。
    4.  **裏数字（横パス）:** グリッドに変化がなくなるまで、`grid[y][x] = (grid[y-1][x] + 5) % 10` を適用。
    5.  **余りマス:** グリッドに変化がなくなるまで、`grid[y][x] = grid[y-1][x]` を適用。
*   **3.6 出力:** 完成した2次元配列の表データ。

#### **4.0 API仕様**
*   **`POST /api/v1/predict/axis`:** 抽選回とリハ数字を送り、軸数字候補のリストを受け取る。
*   **`POST /api/v1/predict/combination`:** 抽選回、リハ数字、選択した軸数字を送り、最終予測リストを受け取る。

### **DOC-03-IMPL：実装計画書 (v1.0)**

#### **1.0 開発戦略：MVPからの段階的アプローチ**
本書で定義された「完全版」のシステムは、非常に高度で多機能である。これを一度に開発しようとすると、莫大な時間とコストを要し、リスクも高い。
そこで、本プロジェクトは**MVP（Minimum Viable Product）**から着手し、段階的に機能を拡張していく、アジャイルな開発アプローチを採用する。

**目的:**
*   **価値の早期検証:** プロジェクトの核心的価値（理論に基づくAI予測）が技術的に実現可能であることを、最短時間で証明する。
*   **リスクの早期発見:** 開発プロセスの全体を一気通貫で体験し、技術的な障壁や設計上の問題点を早期に洗い出す。
*   **モチベーションの維持:** 実際に「動く製品」を早期に手に入れることで、長期的な開発へのモチベーションを維持する。

#### **2.0 MVPのスコープ定義**
*   **目標:** **7日間**で、依頼者が実際に触れる、動くAI予測ツールをデプロイする。
*   **範囲内 (In-Scope):**
    *   **ナンバーズ3** および **ナンバーズ4** の予測機能。
    *   予測表は**「0なし（パターンA）」**のみ。
    *   学習データは**直近100件**のリハーサル数字ありデータに限定。
    *   データ管理は**CSVファイル**で行う（データベースは使用しない）。
    *   開発環境は**ローカルPC**を主とし、最終的に**Vercel**にデプロイする。
*   **範囲外 (Out-of-Scope):**
    *   「0あり（パターンB）」の予測表生成。
    *   過去6000回分の全データを用いた学習、およびアンサンブル学習。
    *   Supabase, GCP Cloud Runの使用。
    *   ユーザー認証、ダッシュボード、履歴管理機能。

#### **4.0 MVP以降の拡張ロードマップ**
MVPの成功後、以下の順で段階的に機能を拡張し、「完全版」を目指す。
1.  **Phase 2 (信頼性向上):** 学習データをリハーサルあり全データ（約1,800件）に拡大し、AIモデルを再学習・更新。
2.  **Phase 3 (アンサンブル導入):** 学習データを過去全データ（約6,700件）に拡大。「チャート・マスターモデル」を構築し、アンサンブル学習を実装。
3.  **Phase 4 (全理論実装):** 「0あり（パターンB）」の表生成ロジックと、それに対応する「統一モデル」を実装。
4.  **Phase 5 (プロフェッショナル化):** データ基盤をCSVからSupabaseデータベースへ移行。AIエンジンをGCP Cloud Runへ分離。OCRやダッシュボードなどの追加機能を開発。
