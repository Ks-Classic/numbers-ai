## 🎯 v2.1の追加内容

### keisen_master.json作成方法ドキュメントの追加

**追加理由:**
- `keisen_master.json`はプロジェクトの核心部分であり、その正確性が予測システム全体の精度を左右する
- 作成方法が不明確なままでは、正確なデータ作成が困難
- 今後のすべての作業の基盤となるため、早期にドキュメント化が必要

**作成内容:**
- ドキュメント構造と枠組みを定義
- 重要性と影響範囲を明確化
- 検証方法の枠組みを定義
- 具体的な作成方法と予測出目ルールは、紙媒体の罫線ルール確定後に追加予定

## 🎯 v2.0の改善点

### 1. カテゴリ別分割

**従来の問題:**
- `requirements.md`: 30,000文字（長すぎる）
- `specifications.md`: 35,000文字（非常に長大）

**v2.0の解決:**
- 7つのカテゴリに分割
- 各ドキュメント6,000〜15,000文字
- 役割別・フェーズ別に最適化

### 2. 開発フェーズごとの最適化

**MVP開発時に読むべきドキュメント:**
1. 06-implementation-plan.md（必読）⭐️
2. 01-business-requirements.md
3. 02-system-architecture.md
4. 04-algorithm-ai.md

**Phase 2移行時に読むべきドキュメント:**
1. 06-implementation-plan.md（Phase 2セクション）
2. 03-data-api-design.md（データベース設計）

### 3. 技術選定理由の明確化

**「なぜ今この技術を使うのか」「なぜ今は使わないのか」**を明文化：
- MVP: なぜSupabaseを使わない → CSV管理で十分
- Phase 2: なぜSupabaseを導入 → データ量1,800件で必要に
- Phase 3: なぜW&Bを導入 → 実験管理の科学化
- Phase 4: なぜCloud Run → Vercelの制限を克服
